# Дизайн: инфраструктура экосистемы AiBox на новом сервере

Дата: 2026-07-03. Статус: принят (вариант A из брейншторма).

## Контекст

Экосистема AiBox переезжает на новую инфраструктуру — пустой сервер, запуск
в Docker. На старом проде приложения работали в `network_mode: host` и
опирались на **хостовые** nginx, MariaDB и Redis; на новом сервере их нет —
весь фундамент нужно привезти с собой.

Переезжают: **ai-box** (Laravel 12, PHP 8.3), **ai-box-data-registry**
(Laravel 12 + Qdrant), **ai-box-mcp** (Laravel 12), **ai-box-front**
(Vite SPA, статика), **ai-box-pdn-cleaner** (Python/FastAPI + BERT-NER,
репозиторий ai-box-bert-ner-train, требует GPU — на новом сервере GPU есть),
**ollama-router** (Go-прокси + пул Ollama-инстансов по одному на GPU; сейчас —
systemd-юниты на хосте, не докер). `ll-service` (Yii2) не переезжает —
остаётся внешней системой.

Проблемы старой схемы: три почти идентичных `Dockerfile.prod` (различие —
только `WORKDIR`), неявная зависимость от хостовых сервисов, конфигурация
входа (nginx/TLS) не версионировалась вместе с приложениями.

## Решение (вариант A): shared-стек + тонкие app-стеки

Два уровня:

1. **`ai-box-infra` (этот репозиторий)** — владеет «фундаментом»:
   - edge-nginx: публичные vhost'ы с TLS (ai-box / api / admin .amulex.ru),
     статика фронта, внутренние vhost'ы DR (8083) и MCP (8084) без
     публикации на хост;
   - MariaDB 11.8 (последний LTS), один инстанс, база+пользователь на
     приложение, создаются init-скриптом;
   - Redis 7.4, один инстанс, DB-индексы распределены по приложениям;
   - Qdrant v1.12.4 (пин версии текущего прода DR);
   - browserless/chrome;
   - docker-сеть `ecosystem` (attachable) — общая шина всех стеков;
   - базовый PHP-образ `aibox/php-base:8.3` — объединённый Dockerfile трёх
     приложений без `WORKDIR`-специфики;
   - certbot: первичная выдача standalone (до старта nginx), продление
     webroot через cron хоста.

2. **Репозитории приложений** — тонкие прод-compose: только `php-fpm`,
   `worker` (и scheduler при необходимости) от `aibox/php-base:8.3`,
   без `network_mode: host`, подключение к внешней сети `ecosystem`.

Контракты (имена контейнеров, пути кода, hostnames, Redis-индексы) — в
[README](../../../README.md).

## Рассмотренные альтернативы

- **B. Один общий compose на всё** — проще поднять сервер одной командой, но
  деплой любого приложения становится операцией над общим стеком и релизы
  связываются. Отклонён.
- **C. Каждый сам за себя** (свои mariadb/redis в каждом прод-compose) —
  быстрее переехать, но 3×MariaDB и 3×Redis на одном сервере, дублирование
  Dockerfile'ов сохраняется, а общий слой (вход/TLS/фронт) всё равно нужен.
  Отклонён.

## Ключевые решения и trade-off'ы

- **Резолв upstream'ов в рантайме** (`resolver 127.0.0.11` + переменная в
  `fastcgi_pass`): nginx стартует и живёт независимо от контейнеров
  приложений. Цена — чуть более многословные vhost'ы.
- **Код приложений — bind-mount с хоста** по одинаковому пути
  `/var/www/<repo>` в nginx (ro) и php-fpm: сохраняет текущую модель
  деплоя «git-клон на хосте» и корректный `SCRIPT_FILENAME`. Деплой-механику
  решено пока не пересматривать.
- **Один Redis с DB-индексами вместо инстанса на приложение**: экономия
  памяти и одна точка обслуживания; изоляция — индексами и паролем. Риск:
  общий лимит памяти — при росте очередей вынести проблемное приложение в
  отдельный инстанс.
- **Внутренние API (DR, MCP) — vhost'ы на непубликуемых портах** shared-nginx
  (`gateway:8083/8084`), а не свои nginx в каждом стеке: меньше контейнеров,
  вся маршрутизация в одном месте. Цена — infra-репо знает о приложениях
  (пути `public/`); принято осознанно, состав экосистемы меняется редко.
- **Базовый образ без реестра**: пока собирается локально на сервере
  (`make build-base`). При появлении CI — публиковать в GitLab Container
  Registry с версионированными тегами.
- **Исправлена опечатка пути** старого прода DR (`ai-box-data-regestry`) —
  на новом сервере канонично `ai-box-data-registry`; учесть в `.env`/конфигах
  приложения при переезде.

## Что меняется в репозиториях приложений (следующие этапы)

1. Новый `docker-compose.prod.yml`: `image: aibox/php-base:8.3` (или
   `FROM` него в 3-строчном Dockerfile), сеть `ecosystem` (external),
   контейнеры `ai-box-php` / `ai-box-dr-php` / `ai-box-mcp-php`,
   `working_dir: /var/www/<repo>`, php-fpm слушает 9000 на сети (не loopback).
2. `.env` прода: `DB_HOST=mariadb`, `REDIS_HOST=redis`, `REDIS_DB`/
   `REDIS_CACHE_DB` по таблице индексов, URL соседей через `gateway`.
3. ai-box: `DEMO_BROWSERLESS_*` → `ws://browserless:3000`; `ll-service` —
   по внешнему адресу (доступность с нового сервера проверить отдельно).
4. ai-box-pdn-cleaner: подключение api-контейнера к сети `ecosystem` под
   именем `ai-box-pdn-cleaner`, убрать собственный `pii-redis` в пользу
   общего Redis (`REDIS_URL=redis://:пароль@redis:6379/6`), порт 8000 наружу
   не публиковать (потребители ходят по сети); GPU-привязка (`GPU_ID`) и
   запечённый в образ код остаются как есть.
5. ollama-router: целевой вариант — докеризация (см. открытые вопросы):
   compose в его репозитории — контейнер роутера `ollama-router:11434` на
   сети `ecosystem` + официальные образы ollama по одному на GPU
   (`device_ids` по UUID, порты 1144x только внутри сети), метрики 9090 —
   на loopback хоста для Zabbix. Потребители конфигурируют `OLLAMA_URL`
   и от варианта размещения не зависят.
6. dev-окружения не трогаем (решение «пока только прод»).

## План переезда (черновик)

1. Подготовить сервер: docker, git-клоны приложений в `${APPS_ROOT}`.
2. `make build-base && make certs-init && make up` (см. README).
3. Прогнать приложения на новых compose против пустых баз (smoke).
4. Перенести данные: mysqldump со старого прода → импорт; Qdrant — снапшот
   коллекций или реиндексация датасетов; Redis не переносим (кэш/очереди —
   дождаться разбора очередей перед выключением старого прода).
5. Переключить DNS доменов на новый сервер, перевыпустить/перенести сертификат.
6. Старый прод — в read-only до подтверждения, затем гасим.

## Открытые вопросы

- **ollama-router: докер или systemd** — рекомендация: докеризовать
  (единая операционная модель на свежем сервере, доступ по имени на сети
  `ecosystem`, не нужно провижининга хоста: пользователь ollama, бинари,
  симлинки юнитов). Цена — переложить systemd-юниты в compose и
  перепроверить GPU-пиннинг по UUID и поведение VRAM/автоскейлера.
  Запасной вариант заложен: подсеть `ecosystem` зафиксирована
  (172.30.0.0/24), при systemd-варианте контейнеры ходят на хостовый
  11434 через gateway 172.30.0.1. Ждёт подтверждения.
- **Вход/TLS**: заложена самостоятельная терминация (Let's Encrypt). Если
  инфра-команда поставит внешний прокси — vhost'ы переводятся на `listen 80`
  без ssl-сниппета, схема не меняется.
- **Деплой**: сознательно вне скоупа; текущая модель (git-клон на хосте,
  Jenkins по SSH) продолжает работать.
- **Ресурсы сервера** (CPU/RAM/диск) неизвестны — `innodb_buffer_pool_size`
  (512M) и лимиты browserless подобрать по факту.
