# Дизайн: инфраструктура экосистемы AiBox на новом сервере

Дата: 2026-07-03. Статус: принят (вариант A из брейншторма).

## Контекст

Экосистема AiBox переезжает на новую инфраструктуру — пустой сервер, запуск
в Docker. На старом проде приложения работали в `network_mode: host` и
опирались на **хостовые** nginx, MariaDB и Redis; на новом сервере их нет —
весь фундамент нужно привезти с собой.

Переезжают: **ai-box** (Laravel 12, PHP 8.3), **ai-box-data-registry**
(Laravel 12 + Qdrant), **ai-box-mcp** (Laravel 12), **ai-box-front**
(Vite SPA, статика), **ai-box-pdn-cleaner** (Python/FastAPI + BERT-NER,
репозиторий ai-box-bert-ner-train, требует GPU — на новом сервере GPU есть),
**ollama-router** (Go-прокси + пул Ollama-инстансов по одному на GPU; сейчас —
systemd-юниты на хосте, не докер). `ll-service` (Yii2) не переезжает —
остаётся внешней системой.

Проблемы старой схемы: три почти идентичных `Dockerfile.prod` (различие —
только `WORKDIR`), неявная зависимость от хостовых сервисов, конфигурация
входа (nginx/TLS) не версионировалась вместе с приложениями.

## Решение (вариант A): shared-стек + тонкие app-стеки

Два уровня:

1. **`ai-box-infra` (этот репозиторий)** — владеет «фундаментом»:
   - edge-nginx: публичные vhost'ы с TLS — новая раскладка доменов:
     `app.` (фронт SPA), `api.`, `admin.`, корневой домен → пока 301-редирект
     на `app.`; статика фронта, внутренние vhost'ы DR (8083) и MCP (8084)
     без публикации на хост;
   - MariaDB 11.8 (последний LTS), один инстанс, база+пользователь на
     приложение, создаются init-скриптом;
   - Redis 7.4, один инстанс, DB-индексы распределены по приложениям;
   - Qdrant v1.12.4 (пин версии текущего прода DR);
   - browserless/chrome;
   - docker-сеть `ecosystem` (attachable) — общая шина всех стеков;
   - базовый PHP-образ `aibox/php-base:8.3` — объединённый Dockerfile трёх
     приложений без `WORKDIR`-специфики;
   - certbot: первичная выдача standalone (до старта nginx), продление
     webroot через cron хоста.

2. **Репозитории приложений** — тонкие прод-compose: только `php-fpm`,
   `worker` (и scheduler при необходимости) от `aibox/php-base:8.3`,
   без `network_mode: host`, подключение к внешней сети `ecosystem`.

Контракты (имена контейнеров, пути кода, hostnames, Redis-индексы) — в
[README](../../../README.md).

## Модели размещения (переходный период)

Поддерживаются две топологии, и какое-то время живут **две копии
приложения одновременно** — одна полностью самодостаточная, одна
разделённая:

1. **«Всё внутри»** — один сервер с GPU: shared-стек, PHP-приложения,
   pdn-cleaner и ollama-пул на одной машине; потребители ходят по именам
   сети `ecosystem` (`OLLAMA_URL=http://ollama-router:11434`,
   `PDN_CLEANER_URL=http://ai-box-pdn-cleaner:8000`).
2. **«Сплит»** — app-сервер без GPU: shared-стек + PHP-приложения + фронт;
   Ollama и pdn-cleaner живут на другой машине. Разница с моделью 1 —
   **только значения env** у потребителей: `OLLAMA_URL`/`PDN_CLEANER_URL`
   указывают на адреса GPU-хоста. Составы compose-стеков не меняются —
   GPU-стеки просто подняты на другом хосте.

Следствия для конструкции:

- **Каждая копия — независимый инстанс infra-стека**: свои volumes
  (данные), свои пароли, свои домены. Домены и имя сертификата
  параметризованы через `.env` (`FRONT_DOMAIN`/`API_DOMAIN`/`ADMIN_DOMAIN`/
  `CERT_NAME`) — nginx-vhost'ы рендерятся из `nginx/templates/*.template`
  механизмом envsubst официального образа. Ничего между копиями не
  шарится; перенос данных между ними — теми же процедурами, что и
  миграция (дампы).
- **Канал до вынесенных GPU-сервисов — только закрытый.** У ollama-router
  нет собственной авторизации, у pdn-cleaner Bearer-токен есть, но через
  него ходят тексты с ПДн. Рекомендация: WireGuard между app- и GPU-хостом,
  на GPU-хосте публиковать 11434/8000 только на VPN-интерфейс
  (`listen <vpn-ip>:port` в ports compose). Минимально допустимая
  альтернатива — firewall-allowlist по IP app-сервера; голым в интернет —
  нельзя.

## Рассмотренные альтернативы

- **B. Один общий compose на всё** — проще поднять сервер одной командой, но
  деплой любого приложения становится операцией над общим стеком и релизы
  связываются. Отклонён.
- **C. Каждый сам за себя** (свои mariadb/redis в каждом прод-compose) —
  быстрее переехать, но 3×MariaDB и 3×Redis на одном сервере, дублирование
  Dockerfile'ов сохраняется, а общий слой (вход/TLS/фронт) всё равно нужен.
  Отклонён.

## Ключевые решения и trade-off'ы

- **Резолв upstream'ов в рантайме** (`resolver 127.0.0.11` + переменная в
  `fastcgi_pass`): nginx стартует и живёт независимо от контейнеров
  приложений. Цена — чуть более многословные vhost'ы.
- **Код приложений — bind-mount с хоста** по одинаковому пути
  `/var/www/<repo>` в nginx (ro) и php-fpm: сохраняет текущую модель
  деплоя «git-клон на хосте» и корректный `SCRIPT_FILENAME`. Деплой-механику
  решено пока не пересматривать.
- **Один Redis с DB-индексами вместо инстанса на приложение**: экономия
  памяти и одна точка обслуживания; изоляция — индексами и паролем. Риск:
  общий лимит памяти — при росте очередей вынести проблемное приложение в
  отдельный инстанс.
- **Внутренние API (DR, MCP) — vhost'ы на непубликуемых портах** shared-nginx
  (`gateway:8083/8084`), а не свои nginx в каждом стеке: меньше контейнеров,
  вся маршрутизация в одном месте. Цена — infra-репо знает о приложениях
  (пути `public/`); принято осознанно, состав экосистемы меняется редко.
- **Базовый образ без реестра**: пока собирается локально на сервере
  (`make build-base`). При появлении CI — публиковать в GitLab Container
  Registry с версионированными тегами.
- **Исправлена опечатка пути** старого прода DR (`ai-box-data-regestry`) —
  на новом сервере канонично `ai-box-data-registry`; учесть в `.env`/конфигах
  приложения при переезде.

## Что меняется в репозиториях приложений

Реализовано 2026-07-03: в каждом репозитории **добавлен** (не заменил
старый прод-файл — его тянет `git pull` деплоя старого прода)
`docker-compose.ecosystem.yml` + `php-fpm-ecosystem.conf` (fpm слушает
9000 на сети) + `eco-*` таргеты Makefile; pdn-cleaner и ollama-router
получили ecosystem-override'ы, ollama-router — Dockerfile и докерный
compose пула. Старые прод-файлы удаляются отдельным коммитом после
завершения переезда.

1. Новый `docker-compose.prod.yml`: `image: aibox/php-base:8.3` (или
   `FROM` него в 3-строчном Dockerfile), сеть `ecosystem` (external),
   контейнеры `ai-box-php` / `ai-box-dr-php` / `ai-box-mcp-php`,
   `working_dir: /var/www/<repo>`, php-fpm слушает 9000 на сети (не loopback).
2. `.env` прода: `DB_HOST=mariadb`, `REDIS_HOST=redis`, `REDIS_DB`/
   `REDIS_CACHE_DB` по таблице индексов, URL соседей через `gateway`.
3. ai-box: `DEMO_BROWSERLESS_*` → `ws://browserless:3000`; `ll-service` —
   по внешнему адресу (доступность с нового сервера проверить отдельно).
   Смена раскладки доменов (фронт: корень → `app.`) тянет конфиги
   приложения: `FRONTEND_URL`/CORS/Sanctum stateful domains, домен админки
   Filament, публичные URL виджетов; в ai-box-front — базовый URL API.
4. ai-box-pdn-cleaner: подключение api-контейнера к сети `ecosystem` под
   именем `ai-box-pdn-cleaner`, убрать собственный `pii-redis` в пользу
   общего Redis (`REDIS_URL=redis://:пароль@redis:6379/6`), порт 8000 наружу
   не публиковать (потребители ходят по сети); GPU-привязка (`GPU_ID`) и
   запечённый в образ код остаются как есть.
5. ollama-router: **докеризация (решено 2026-07-03)** — compose в его
   репозитории: контейнер роутера `ollama-router:11434` на сети `ecosystem`
   + официальные образы ollama по одному на GPU (`device_ids` по UUID,
   порты 1144x только внутри сети), метрики 9090 — на loopback хоста для
   Zabbix. systemd-юниты уходят; их роль (пул + автостарт) берёт compose
   с `restart: unless-stopped`. Потребители конфигурируют `OLLAMA_URL` и
   от варианта размещения не зависят.
6. dev-окружения не трогаем (решение «пока только прод»).

## План переезда

1. Подготовить сервер: docker + NVIDIA Container Toolkit, git-клоны
   приложений в `${APPS_ROOT}`.
2. `make build-base && make certs-init && make up` (см. README; если DNS ещё
   не переключён — вместо `certs-init` перенести каталог `/etc/letsencrypt`
   старого сервера в volume `letsencrypt`, сертификат тот же).
3. Поднять app-стеки, прогнать против пустых баз (smoke: /up, админка,
   внутренние vhost'ы).
4. Миграция данных (см. следующую секцию).
5. Переключить DNS доменов на новый сервер; убедиться, что продление
   сертификата работает (`make certs-renew`).
6. Старый прод — в maintenance/read-only до подтверждения, затем гасим.

## Запуск: сплит-схема на ai-box.amulex.ru (in-place)

Первый боевой запуск — **на текущем сервере**: `ai-box.amulex.ru` остаётся
на месте (DNS не трогается), под приложениями меняется фундамент — хостовые
nginx/MariaDB/Redis → докерный shared-стек + тонкие app-стеки. Топология —
сплит: Ollama-пул и pdn-cleaner **на этом этапе не переключаются вовсе** —
работают как сейчас (systemd-пул, standalone), приложения ходят на их
текущие адреса; их докеризация (заготовки в репозиториях) — более поздний
этап, к модели «всё внутри». Требование — бесшовно, без даунтайма.

Принцип: обе инфры живут на одной машине параллельно — новый стек
поднимается на альтернативных портах (`docker-compose.transition.yml`:
nginx 127.0.0.1:8081/8444, mariadb 3307, qdrant 6334; серты — bind
хостового /etc/letsencrypt), проверяется на реальных данных; **момент
переключения — reload хостового nginx с vhost'ами-прокси на докерный
nginx** (двойной TLS на loopback — осознанная цена неизменности шаблонов);
мгновенный откат тем же reload'ом. Боевые 80/443 стек забирает позже
отдельным секундным шагом, тогда же гасятся хостовые сервисы. Для новых
стеков делаются свежие клоны под каноничными путями — старые каталоги
(`ai-box-back`, `ai-box-data-regestry`) продолжают обслуживать прод до
переключения. Пошагово, с окнами отката и граблями —
[runbook](../runbooks/split-cutover-ai-box.md).

## Миграция данных

Что переносим и как; источник — старый прод (хостовые сервисы + контейнеры).

- **MariaDB** (главное): на старом сервере `mysqldump --single-transaction
  --routines --triggers` по каждой базе (имена/креды — из прод `.env`
  приложений); перенос на новый сервер; импорт `make db-import DB=ai_box
  FILE=ai_box.sql` (аналогично `ai_box_dr`, `ai_box_mcp`). Базы и
  пользователи уже созданы init-скриптом. Проверка: счётчики строк ключевых
  таблиц до/после, `php artisan migrate:status` в каждом приложении.
- **Qdrant** (данные DR): версия пиннута той же (v1.12.4), поэтому
  допустимо простое копирование: остановить старый контейнер qdrant →
  rsync содержимого volume `qdrant_data` в одноимённый volume нового стека →
  поднять. Альтернатива без остановки — снапшоты коллекций через API.
  Запасной путь — реиндексация датасетов средствами DR.
- **Файлы приложений**: `storage/app` каждого приложения (документы
  датасетов DR — обязательно), `.env` прода (с правкой хостов на
  `mariadb`/`redis`/`gateway`-контракты и Redis DB-индексов по таблице).
- **Ollama-модели**: со старого сервера снять список (`/api/tags`) и на
  новом выполнить `ollama pull` по списку в volume пула (чистый путь);
  если есть кастомные/тяжёлые модели или узкий канал — rsync блобов
  `~ollama/.ollama` в volume контейнеров.
- **Модель pdn-cleaner**: не переносим руками — `make model-pull`
  (HF_MODEL_REPO) в его репозитории.
- **Redis не переносим**: кэш прогреется сам; очереди — перед снятием дампа
  включить maintenance, дождаться разбора очередей на старом проде
  (`queue:work` до пустой очереди), потом дампить БД. Сессии/выданные
  Sanctum-токены живут в MariaDB и переезжают с дампом.
- **Порядок в даунтайне**: maintenance на старом → дренаж очередей → дампы
  (MariaDB, qdrant, storage) → перенос → импорт → smoke на новом → DNS.
  До переключения DNS сверить объёмы заранее и гонять rsync инкрементально,
  чтобы окно даунтайна было минутами, а не часами.

## Открытые вопросы

- **Вход/TLS**: заложена самостоятельная терминация (Let's Encrypt). Если
  инфра-команда поставит внешний прокси — vhost'ы переводятся на `listen 80`
  без ssl-сниппета, схема не меняется.
- **Деплой**: сознательно вне скоупа; текущая модель (git-клон на хосте,
  Jenkins по SSH) продолжает работать.
- **Ресурсы сервера** (CPU/RAM/диск) неизвестны — `innodb_buffer_pool_size`
  (512M) и лимиты browserless подобрать по факту.
