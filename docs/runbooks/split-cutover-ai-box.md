# Runbook: запуск сплит-схемы на ai-box.amulex.ru без даунтайма

Цель: перевести прод `ai-box.amulex.ru` (и `api.`/`admin.`, новая раскладка
с `app.`) со старого сервера на новый app-сервер в **сплит-топологии**:
приложения + shared-стек на новом сервере, Ollama-пул и pdn-cleaner остаются
на своих текущих машинах.

**Принцип бесшовности:** не ждать DNS. Момент переключения — reload nginx
на старом сервере в режим reverse-proxy на новый. С этой секунды весь
трафик (включая закэшированный DNS) обслуживает новая копия; DNS
переключается после, в спокойном режиме. Split-brain исключён: старые
upstream'ы отключаются тем же reload'ом.

## Фаза 0 — подготовка нового сервера (без влияния на прод)

1. Docker, git-клоны приложений в `${APPS_ROOT}`, клон этого репо.
2. `.env` стека: домены `ROOT_DOMAIN=ai-box.amulex.ru`,
   `FRONT_DOMAIN=app.ai-box.amulex.ru`, `API_DOMAIN=api.…`, `ADMIN_DOMAIN=admin.…`.
3. Закрытый канал до GPU-хостов (WireGuard/allowlist); в `.env` приложений
   `OLLAMA_URL`/`PDN_CLEANER_URL` — на их текущие адреса. Проверить
   доступность оттуда же: `curl <OLLAMA_URL>/api/tags`, `<PDN>/health`,
   а также ll-service.
4. `make build-base && make up` + app-стеки. Сертификата ещё нет — nginx
   поднимется после фазы 1; smoke до тех пор — по http-портам изнутри сети.
5. Фронт: собрать `ai-box-front/dist` на новом сервере (вручную или
   пайплайном), проверить, что раздаётся.

## Фаза 1 — сертификат до переключения DNS

HTTP-01 на новом сервере невозможен, пока DNS смотрит на старый. Поэтому:

1. DNS: завести `app.ai-box.amulex.ru` → **старый** сервер (нового домена
   ещё нет в проде — безопасно).
2. На старом сервере расширить существующий сертификат:
   `certbot certonly --expand -d ai-box.amulex.ru -d app.ai-box.amulex.ru
   -d api.ai-box.amulex.ru -d admin.ai-box.amulex.ru` (webroot старого nginx).
3. `rsync -a /etc/letsencrypt/ <новый>:…` → в volume `letsencrypt` стека
   (имя lineage = `ai-box.amulex.ru` = дефолтный `CERT_NAME`).
4. `make up` на новом — nginx стартует с валидным сертификатом.
5. Продление до переключения DNS остаётся на старом сервере; после — фаза 4.

## Фаза 2 — данные: начальная загрузка и инкременты

1. Первичный перенос без остановки прода: дампы MariaDB
   (`--single-transaction`, консистентно, прод не блокирует) → `make
   db-import`; `rsync -a` storage/ приложений; volume Qdrant — снапшоты
   коллекций через API (без остановки старого qdrant).
2. Smoke новой копии на реальных данных: `/etc/hosts` на рабочей машине →
   новый IP, пройти фронт/API/админку/виджет.
3. Инкременты storage гонять `rsync`-ом повторно (дёшево); данные БД на
   финальном шаге переносятся заново коротким дампом (см. фазу 3) —
   к этому моменту процедура уже отрепетирована и хронометрирована.
   Если финальный дамп по замерам длиннее допустимой паузы записи —
   поднять репликацию old→new заранее (binlog на старом, replica —
   контейнер mariadb; тогда фаза 3 сжимается до `STOP REPLICA` + promote).

## Фаза 3 — переключение (окно: минуты, чтение не прерывается)

Подготовить заранее на старом сервере proxy-конфиги vhost'ов (те же
`server_name`, `proxy_pass https://<новый-IP>` с `proxy_set_header Host
$host`, `X-Forwarded-*`; для `/up` — без буферизации не требуется).

1. Пауза **записи**: на старом `php artisan down` для api/admin (фронт-статика
   продолжает отдаваться), остановить queue-воркеры после дренажа
   (`queue:work --stop-when-empty` или дождаться пустых очередей).
2. Финальный синк: короткий дамп изменившихся БД → импорт на новом;
   финальный `rsync` storage; снапшот Qdrant, если менялся.
3. На новом: `php artisan migrate --force` (если есть новые миграции),
   прогреть (`/up`, логин, шаг чата).
4. **Момент переключения**: на старом сервере подменить vhost'ы ai-box на
   proxy-конфиги, `nginx -t && nginx -s reload`. Весь трафик уходит на
   новую копию. Старые php-fpm/воркеры ai-box больше не получают запросов —
   остановить их контейнеры.
5. Проверки: фронт, логин (сессии/токены приехали с БД), шаг чата
   (ollama через сплит-канал), маскирование (pdn-cleaner), загрузка
   документа (DR+Qdrant), админка, вебхуки ботов (см. «Грабли»).

Откат (пока живёт proxy-режим): вернуть старые vhost'ы reload'ом. Честное
окно отката — до первых записей в новую БД; после — откат только с обратным
переносом дельты.

## Фаза 4 — DNS и добивка

1. Понизить TTL заранее (до фазы 3), после переключения — A-записи
   root/app/api/admin → новый сервер.
2. Когда трафик на старом прокси иссяк (логи), погасить прокси-vhost'ы.
3. Продление сертификата переносится на новый сервер: `make certs-renew`
   (webroot уже локальный), убрать renewal со старого из cron.
4. Cron на новом: `certs-renew`; бэкапы MariaDB/Qdrant.
5. Старый сервер: контейнеры ai-box остановлены, данные не удалять до
   истечения контрольного срока.

## Грабли, проверить отдельно

- **Вебхуки ботов** (Telegram/Max) сейчас ходят через публичный прокси
  `addons.amulex.ru/ai-box/` — его upstream должен указать на новый сервер
  (или продолжать на старый nginx, пока тот в proxy-режиме — тогда ничего
  не менять до фазы 4).
- **Виджеты на сайтах клиентов**: URL зашиты у клиентов. Пока фронт-бандлы
  доступны по старым путям (корень → 301 на `app.` сохраняет путь) — не
  ломаются; embed-код новых клиентов публиковать уже с `app.`
- **ll-service**: новый сервер должен достучаться (firewall/allowlist);
  проверяется в фазе 0.
- **Jenkins-деплой** фронта и приложений целится в старый сервер —
  перенастроить цели после фазы 3 (деплой в окно фаз 3–4 не запускать).
- Прод `.env` приложений на новом сервере — новые хосты (`mariadb`,
  `redis` + DB-индексы), старые секреты (APP_KEY обязательно тот же, иначе
  расшифровка шифрованных полей сломается).
