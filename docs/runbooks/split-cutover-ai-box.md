> **Статус: ВЫПОЛНЕН на бою 2026-07-04.** Прод ai-box.amulex.ru переведён
> на докерный стек по этому runbook'у: окно записи 76 сек (дампы+rsync),
> данные сошлись точь-в-точь (chat_sessions 186275=186275, Qdrant 6
> коллекций без дельты), переключение apply.sh (прокси-vhost'ы+reload),
> живой клиентский трафик пошёл сразу (create/step 200). Старые
> app-контейнеры остановлены; хостовые MariaDB/Redis и старый qdrant —
> в контрольном сроке. Осталась фаза 3 (захват 80/443, cron certs-renew,
> Jenkins на новые каталоги, вывод хостовых сервисов).
>
> Боевые отклонения от плана: (1) MCP делит каталог со старым стеком и
> имеет config-кэш → свежий клон `/var/www/ai-box-mcp-eco` + некоммитный
> `docker-compose.prod-local.yml` (nginx-mount на него); (2) `REDIS_PASSWORD=null`
> у старого прода → новому Redis выдан настоящий пароль (обновить в 4 .env);
> (3) Redis DB-индексы оставлены прод-историческими (ai-box 0/1, DR 6/7,
> MCP 8/9), а не по таблице README; (4) compose v2.12 → обновлён до v2.27
> через apt (deb-конфликт починен), `!override` заработал; (5) образ
> curlimages/curl с боя не тянется — сетевые проверки делать `docker exec`
> в ai-box-php (curl в базовом образе есть).
>
> Метод предварительно обкатан на dev-машине 2026-07-03 — dev-уроки
> помечены «(dev-урок)» в фазах ниже.

Сервер **остаётся тот же** (`ai-box.amulex.ru`), DNS не переключается.
Меняется фундамент под приложениями: хостовые nginx/MariaDB/Redis и
россыпь per-app контейнеров в `network_mode: host` → докерный shared-стек
`ai-box-infra` + тонкие app-стеки на сети `ecosystem`. Топология —
**сплит**: Ollama-пул и pdn-cleaner остаются на своих машинах, приложения
ходят к ним по `OLLAMA_URL`/`PDN_CLEANER_URL` через закрытый канал.
Заодно вводится новая раскладка доменов (`app.` + корень → редирект).

**Вне скоупа этого этапа:** pdn-cleaner и ollama-router не переключаются —
работают как сейчас (systemd-пул Ollama, standalone pdn-cleaner), приложения
продолжают ходить на их текущие адреса. Их докеризация/ecosystem-режим
(заготовки уже в репозиториях) — отдельный, более поздний этап.

**Принцип бесшовности:** обе инфры живут на одной машине параллельно.
Новый стек поднимается на альтернативных портах
(`docker-compose.transition.yml`), проверяется на реальных данных, затем
хостовый nginx одним reload'ом превращается в тонкий прокси перед докерным
nginx — это момент переключения (мгновенный откат тем же reload'ом).
Боевые порты 80/443 стек забирает позже, отдельным коротким шагом.

## Фаза 0 — подготовка (прод не трогаем)

1. Свежие клоны приложений под каноничными именами контрактов:
   `/var/www/ai-box`, `/var/www/ai-box-data-registry`, `/var/www/ai-box-mcp`
   (+ `dist` фронта). Старые каталоги (`ai-box-back`,
   `ai-box-data-regestry` — с опечаткой) не трогаем: из них работает
   текущий прод. Диск дешевле бесшовности.
2. Клон этого репо, `.env`: `ROOT_DOMAIN=ai-box.amulex.ru`,
   `FRONT_DOMAIN=app.ai-box.amulex.ru`, `API_DOMAIN`/`ADMIN_DOMAIN`,
   `APPS_ROOT=/var/www`, пароли новой MariaDB/Redis.
   (dev-урок) Проверить на хосте занятость: альт-портов transition-файла
   (`ss -tlnp`; на dev 8081 оказался занят) и подсети сети ecosystem
   (`ip route`; на dev вся 172.17–31 была выедена) — при пересечении задать
   `ECOSYSTEM_SUBNET`/`ECOSYSTEM_GATEWAY` в `.env`. Сеть ecosystem создаёт
   первый `up` инфра-стека — руками `docker network create` не делать.
   (dev-урок) Версию Qdrant задать в `.env` (`QDRANT_VERSION`) — цель обязана
   быть >= фактической версии источника данных (проверить `GET /` на
   работающем qdrant-источнике); даунгрейд storage не поднимется.
3. GPU-сервисы не переключаем: в `.env` новых клонов `OLLAMA_URL`/
   `PDN_CLEANER_URL` — те же адреса, что в старом проде. Проверить из
   нового стека `curl <OLLAMA_URL>/api/tags` и `<PDN_CLEANER_URL>/health`
   (канал уже существует — сервисы и сейчас используются отсюда; единственный
   нюанс — контейнеры ходят с адресов подсети 172.30.0.0/24, при
   firewall-allowlist по IP это тот же внешний IP сервера, менять нечего).
4. DNS: добавить `app.ai-box.amulex.ru` → **этот же IP**. На хостовом
   certbot расширить сертификат: `certbot certonly --expand -d
   ai-box.amulex.ru -d app.… -d api.… -d admin.…` (webroot хостового nginx).
5. `make build-base`; поднять стек с overlay:
   `docker compose -f docker-compose.yml -f docker-compose.transition.yml up -d`
   — nginx на 127.0.0.1:8090/8444 (серты — bind хостового /etc/letsencrypt),
   MariaDB на 127.0.0.1:3307, qdrant на 6334 (боевые порты заняты старой
   инфрой). Поднять новые app-стеки (сеть `ecosystem`, портов на хост не
   публикуют — с старыми контейнерами не конфликтуют).
6. `.env` приложений в новых клонах: `DB_HOST=mariadb`, `REDIS_HOST=redis`
   (+ **обязательно** `REDIS_DB`/`REDIS_CACHE_DB` по таблице README — без
   них все приложения сядут в DB 0/1 и перемешают очереди), `OLLAMA_URL`/
   `PDN_CLEANER_URL` — как сейчас, **`APP_KEY` — тот же, что в старом
   проде** (шифрованные поля). (dev-урок) Межсервисные URL: в ai-box
   `DATA_REGISTRY_URL=http://gateway:8083`, `AIBOX_MCP_URL=http://gateway:8084`;
   в MCP `AIBOX_BASE_URL=http://gateway:8085` (внутренний vhost ai-box —
   добавлен по итогам dev-миграции); browserless — `http://browserless:3000`.
7. (dev-урок) Перед стартом eco-стеков выровнять владельца изменяемых
   каталогов: `chown -R 1001:1001 storage bootstrap/cache` в каждом
   приложении (старые стеки могли писать под другим uid — на dev
   `laravel.log` оказался root'овым и всё легло 500-ками).

## Фаза 1 — данные: начальная загрузка и репетиция

Всё локально (одна машина) — быстро и без сети.

**Чек-лист переносимых данных** (полный; повторяется в финальном окне фазы 2):

| Данные | Откуда | Куда | Как |
|---|---|---|---|
| БД `ai_box` | хостовая MariaDB | mariadb инфры | mysqldump → db-import |
| БД `ai_box_dr` | хостовая MariaDB | mariadb инфры | mysqldump → db-import |
| БД `ai_box_mcp` | хостовая MariaDB | mariadb инфры | mysqldump → db-import |
| `storage/` ai-box | /var/www/ai-box-back | /var/www/ai-box | rsync -a |
| `storage/` DR (документы датасетов!) | /var/www/ai-box-data-regestry | /var/www/ai-box-data-registry | rsync -a |
| `storage/` MCP | старый каталог MCP | /var/www/ai-box-mcp | rsync -a |
| Коллекции Qdrant | контейнер ai_box_dr_qdrant (volume) | qdrant инфры | снапшоты API или копия volume |
| `.env` всех трёх | старые каталоги | новые клоны | вручную, с правкой хостов |

Не переносится: Redis (кэш прогреется, очереди дренируются), данные
GPU-сервисов (на этом этапе не переключаются).

1. Дампы из хостовой MariaDB: `mysqldump --single-transaction --routines
   --triggers` **каждой из трёх баз** → `make db-import` в контейнерную
   (порт 3307 или `exec`). Прод не блокируется.
2. `rsync -a` storage/ из старых каталогов в новые клоны — **всех трёх
   приложений** (для DR это загруженные документы датасетов — потеря
   невосполнима, реиндексация Qdrant их не вернёт).
3. Qdrant: снапшоты **каждой коллекции** старого контейнера (API :6333,
   `GET /collections` — список) → restore в новый (:6334); или в финальном
   окне — копия volume при остановленных qdrant'ах (версии одинаковые
   v1.12.4, локально — секунды). Volume старого контейнера не удалять
   (`down` без `-v`!) до конца контрольного срока фазы 3.
4. Проверка полноты: счётчики строк ключевых таблиц в каждой базе
   (`SELECT COUNT(*)`) до/после; `points_count` каждой коллекции Qdrant
   (`GET /collections/<name>`) до/после; размер/число файлов storage
   (`du -s`, `find | wc -l`).
5. Smoke новой копии: `curl -k --resolve app.ai-box.amulex.ru:8444:127.0.0.1
   https://app.ai-box.amulex.ru:8444/...` (и api/admin): фронт, логин, шаг
   чата (сплит-канал до Ollama), маскирование, документ DR (и поиск по
   существующему датасету — проверяет перенос Qdrant), админка.
6. Отхронометрировать финальный дамп на репетиции. Если дольше допустимой
   паузы записи — поднять репликацию host-MariaDB → контейнер заранее
   (binlog на хостовой, если не включён — включение требует её рестарта:
   планировать отдельно), тогда финал = `STOP REPLICA` + promote.

## Фаза 2 — переключение (пауза только на запись, минуты)

Заранее подготовить прокси-версии vhost'ов хостового nginx: те же
`server_name`, вместо `root`/`fastcgi_pass` —
`proxy_pass https://127.0.0.1:8444` c `proxy_ssl_server_name on;
proxy_ssl_name $host; proxy_set_header Host $host;` и `X-Forwarded-*`
(двойной TLS на loopback — осознанно: шаблоны стека не меняются, приложения
видят https; серты одни и те же). Плюс новый vhost `app.` (прокси туда же)
и редирект с корня.

1. Пауза записи в старом проде: `php artisan down` в старых ai-box/DR/MCP,
   воркеры — дренаж до пустых очередей и стоп.
2. Финальный синк — по чек-листу фазы 1 целиком: дампы **всех трёх баз** →
   импорт; финальный `rsync` storage всех трёх приложений; Qdrant —
   снапшот/копия volume, если менялся с фазы 1. Прогнать проверки полноты
   (п. 4 фазы 1).
3. В новых app-стеках: `php artisan migrate --force`, прогрев (`/up`, логин).
4. **Переключение**: подменить конфиги хостового nginx на прокси-версии,
   `nginx -t && systemctl reload nginx`. Весь трафик — в новый стек.
5. Проверки под живым трафиком (список из фазы 1.4 + вебхуки ботов).
6. Старые app-контейнеры остановить (запросов они уже не получают).

**Откат**: вернуть прежние конфиги хостового nginx + `reload`, старые
контейнеры `up`, `artisan up`. Честное окно — до первых записей в новую
БД; дальше откат требует обратного переноса дельты.

## Фаза 3 — вывод старой инфры (короткий шаг, секунды)

Когда новая копия отстоялась (дни):

1. Скопировать `/etc/letsencrypt` хоста в volume `letsencrypt` стека.
2. Остановить хостовый nginx и перезапустить стек без transition-файла
   (nginx забирает 80/443, серты из volume):
   `systemctl stop nginx && docker compose up -d` — разрыв на секунды;
   делать в тихое окно. `systemctl disable nginx`.
3. Продление сертов: cron `make certs-renew` (webroot теперь докерный),
   хостовый certbot-таймер выключить.
4. Хостовые MariaDB/Redis: неделя в режиме «работают, но без клиентов» →
   `systemctl disable`, данные не удалять до контрольного срока.
5. Старый qdrant-контейнер погасить; при желании убрать override-порт
   6334→6333 (перезапуск стека).
6. Старые каталоги (`ai-box-back`, `ai-box-data-regestry`) — в архив после
   контрольного срока; Jenkins-цели деплоя перевести на новые пути/стеки.

## Грабли

- **Вебхуки ботов** через `addons.amulex.ru/ai-box/`: сервер и порт не
  меняются — вебхуки не трогаем вовсе (проверить в фазе 2.5).
- **Виджеты у клиентов** (проверено на dev): embed-сниппеты генерятся в
  кабинете от `window.location.origin` — после переезда кабинета на `app.`
  новые сниппеты сами получат новые URL. Старые embed'ы
  (`<корень>/widget/chat-assistant.js`) работают через 301 с корня —
  браузер следует редиректу для `<script src>` (проверено: 200 через
  редирект). `data-api-url` в сниппетах зашит на `api.`-домен — он не
  меняется. Нюанс: у самого клиента виджета дефолт
  `https://addons.amulex.ru/llm/assistant` — используется только если
  embed без `data-api-url`; таких клиентов при выключении addons-прокси
  проверить отдельно.
- **CORS/фронт при смене раскладки**: в прод `.env` ai-box —
  `CORS_ALLOWED_ORIGINS` += `https://app.ai-box.amulex.ru` (иначе SPA не
  залогинится); фронт пересобирать на прод не нужно (`.env.production`
  уже целится в `api.`-домен, он не меняется). Локальная сборка фронта:
  vite в build-режиме берёт `.env.production`, поэтому локальный URL —
  только через env-override: `VITE_API_BASE_URL=https://api.ai-box.local/api
  npm run build`.
- **`APP_KEY` и прочие секреты** — копируются из старых `.env` как есть;
  меняются только хосты подключений.
- **Порты внутренних API**: старый прод держит DR/MCP на хостовых
  127.0.0.1:8083/8084 (хостовый nginx) — у докерного nginx те же номера,
  но внутри сети `ecosystem`, на хост не публикуются: конфликта нет, но в
  `.env` новых приложений URL соседей — `http://gateway:8083/8084`, не
  127.0.0.1.
- **Jenkins**: деплой в окно фаз 2–3 не запускать; после — перенастроить
  на новые каталоги/стеки.
- Хостовая MariaDB могла жить с иными sql_mode/charset-дефолтами — smoke
  фазы 1.4 на реальных данных это ловит; при расхождениях фиксировать
  настройками в `mariadb/my.cnf`, а не правкой данных.
