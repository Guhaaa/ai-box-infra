description: Обновление вики проекта по дельте кода (после правок, сделанных вне Claude Code)
---

Обнови `.claude/wiki/` по изменениям в коде. Воркфлоу, структура и конвенции —
`.claude/rules/wiki.md`. Этим документом руководствуйся при выборе папки,
названии страницы, оформлении frontmatter и перекрёстных ссылок.

Команда применяется в кейсе, когда правки кода уже сделаны (часто — самим
пользователем через IDE/терминал), а вика отстала. Назначение — закрыть
этот разрыв.

## Шаг 1. Определи дельту

Источники изменений по убыванию приоритета:

1. **Аргумент команды** — если задан ref-range (`HEAD~5..HEAD`) или список
   путей, бери только их.
2. **Незакоммиченные изменения** — `git status --porcelain` + `git diff` и
   `git diff --cached`.
3. **Коммиты, не отражённые в вике** — диапазон от последнего коммита,
   тронувшего `.claude/wiki/`, до `HEAD`. Найди его так:
   ```
   last_wiki=$(git log -n 1 --pretty=format:%H -- .claude/wiki)
   git log --name-status "$last_wiki..HEAD"
   ```
   Игнорируй коммиты с трейлером `Wiki: skip` в сообщении.

Из получившейся дельты исключи: `vendor/`, `node_modules/`, `data/`,
`.beads/`, лок-файлы, артефакты сборки. Сосредоточься на исходниках и
конфигурации.

## Шаг 2. Определи затронутые сущности

Для каждого изменённого файла определи, к какой сущности вики он относится:

- **Модули** — папки `modules/<name>/` в коде → страницы `.claude/wiki/modules/`.
- **Интеграции** — клиенты внешних API, очереди, S3, БД-провайдеры →
  `.claude/wiki/integrations/`.
- **Слои/сущности** — модели, контроллеры, миграции, фабрики данных,
  общие компоненты → `.claude/wiki/entities/`.
- **Сквозные темы** — конвенции, паттерны, инфраструктура, тесты →
  `.claude/wiki/concepts/`.
- **Архитектурные решения** — крупная развилка, изменение подхода,
  введение нового слоя/инструмента → `.claude/wiki/decisions/`.

Поиск релевантных страниц:

1. По `sources:` в frontmatter — `grep -l '<затронутый-путь>'
   .claude/wiki/**/*.md` (с учётом расширений).
2. По slug'у и названию — имя модуля/интеграции часто совпадает с именем
   страницы.
3. По ссылкам в `index.md` и `overview.md` — если страница не найдена,
   возможно, её ещё нет.

## Шаг 3. Обнови или создай страницы

Для каждой релевантной страницы:

- Перепиши затронутые разделы под актуальное состояние кода. Истина — в
  коде. Если в вике осталось то, чего в коде уже нет, — убери.
- Обнови `updated:` во frontmatter на сегодняшнюю дату.
- При необходимости дополни `sources:` новыми путями.
- Поставь недостающие `[[тип:slug]]`-ссылки на смежные страницы.
- Если страница разрослась — раздели по принципу «одна сущность — одна
  страница» (см. правило).

Если затронут новый модуль/интеграция/концепция, страницы которой ещё
нет, — создай её. Имя файла — kebab-case без префиксов (папка задаёт тип).
Frontmatter обязателен.

## Шаг 4. Подровняй каталог и журнал

- **`index.md`** — добавь новые страницы в соответствующие разделы; удали
  ссылки на страницы, которые ты убрал.
- **`overview.md`** — если изменился общий ландшафт (новый модуль/слой),
  обнови обзор.
- **`log.md`** — допиши запись:
  ```
  ## [ГГГГ-ММ-ДД] ingest | <краткий заголовок изменений>
  ```

## Шаг 5. Связь с beads

Если правки делались в рамках конкретной задачи beads:

- В метаданных задачи проставь/дополни `wiki_refs` ссылками на затронутые
  страницы:
  ```
  bd update <id> --metadata '{"wiki_refs":["modules/x.md","decisions/y.md"]}'
  ```
- На страницах, появившихся при закрытии решения, добавь секцию
  `## Связанные Beads` с `[[bead:<id>]]`.

## Шаг 6. Отчёт

Выведи короткий отчёт:

- Какие страницы обновлены и почему (1 строка на страницу).
- Какие страницы созданы.
- Что в `index.md`/`overview.md` поменялось.
- Спорные моменты, где ты не уверен и оставил `TODO`.

После этого можно прогнать `/wiki-lint` — он добьёт целостность (битые
ссылки, индекс, синхронизация с beads).
