# Вики проекта

В репозитории ведётся вики — компилируемая база знаний о проекте: markdown-страницы
в `.claude/wiki/`. Вика — часть этого репозитория: версионируется и едет вместе с
кодом в той же ветке. Отдельных веток вики, маркеров синхронизации и догоняющего
Ingest нет.

Вика связана с трекером задач **beads** (`bd`): задача отвечает на «что и когда»,
вика — на «почему и какой ценой». Связь двусторонняя — см. ниже.

## Структура

```
.claude/wiki/
  index.md       каталог всех страниц
  log.md         хронология изменений вики (append-only)
  overview.md    общий обзор проекта
  decisions/     архитектурные решения и trade-off'ы (ядро связи с beads)
  modules/       Yii-модули проекта
  integrations/  внешние системы и API
  entities/      слои приложения и его составные части
  concepts/      паттерны, конвенции, сквозные темы
  raw/           неизменяемые исходники (PRD, скриншоты)
  active-beads/  страницы под крупные задачи (опционально)
```

## Конвенции страниц

- Имя файла — kebab-case, латиница: `style-assistant.md`. Папка задаёт тип —
  префиксы вроде `module-`/`service-` в имени не нужны. Одна страница — одна
  сущность/концепция/решение; разрослась — дроби.
- Каждая страница начинается с YAML-frontmatter:
  ```yaml
  ---
  title: Человекочитаемый заголовок
  type: decision | module | integration | entity | concept | overview
  tags: [module, llm, infrastructure]
  sources: [README.md, services/Llm]
  updated: 2026-05-22
  ---
  ```
- Язык всех страниц, индекса и журнала — русский.

### Конвенции ссылок

Перекрёстные ссылки — в неймспейс-форме `[[тип:slug]]`, где тип = папка:

- `[[module:admin]]`, `[[integration:llm]]`, `[[concept:ai-box]]`,
  `[[entity:data-layer]]`, `[[decision:<slug>]]` — внутри вики; `[[тип:slug]]`
  резолвится в файл `<тип-папка>/<slug>.md`;
- `[[overview]]` — на корневой обзор (корневые страницы не неймспейсятся);
- `[[bead:map-xxxxxx]]` — на задачу beads (id в проекте — `map-<hash>`);
- `raw/<файл>` — на неизменяемый исходник;
- ссылки на код — относительным путём от корня репозитория.

### Секции страниц

- `## Связи` — внутривиковые `[[ссылки]]` (обязательная, как и раньше).
- `## Связанные Beads` — список `[[bead:…]]` на задачи, в рамках которых
  страница менялась; опциональна, добавляется, когда связь появляется.
- `## Источники` — ссылки `raw/<файл>` на неизменяемые исходники; опциональна.

## Двусторонняя связь beads ↔ вика

**bead → wiki.** Ссылки на вику хранятся в метаданных задачи — ключ `wiki_refs`
в `--metadata`, пути относительно `.claude/wiki/`:

```
bd create "Заголовок" --metadata '{"wiki_refs":["decisions/llm-timeouts.md","integrations/llm.md"]}'
bd update map-xxxxxx --metadata '{"wiki_refs":[...]}'
```

**wiki → bead.** На странице — секция `## Связанные Beads` с `[[bead:…]]`-ссылками.

## Воркфлоу жизненного цикла задачи

### Создание задачи

1. Свериться с вики: `index.md`, `decisions/`, профильная папка
   (`modules/`/`integrations/`/`entities/`/`concepts/`). Поиск — `Grep`/`Glob`
   по `.claude/wiki/`.
2. Если нужного знания в вике нет — сперва завести/обновить страницу.
3. Проставить `wiki_refs` в метаданные bead на релевантные страницы.

Тривиальные задачи (опечатки, мелкие правки) сверки не требуют.

### Работа над задачей

1. Прочитать все страницы из `wiki_refs` задачи.
2. Истина — в коде. Если вика противоречит актуальному коду, верь коду и
   исправь вику.
3. Находки и принятые по ходу решения сразу фиксировать в вике — не только в
   коде, в той же ветке и в том же объёме работы.
4. Обновлять поле `updated` во frontmatter изменённых страниц; проставлять и
   проверять перекрёстные ссылки `[[...]]`; обновлять `index.md` при
   добавлении/удалении страниц.

### Закрытие задачи

1. Создать или обновить страницу в `decisions/` (или в профильной папке) с:
   почему выбрано это решение; рассмотренные альтернативы; trade-off'ы и риски;
   уроки; `[[bead:…]]`-ссылки на связанные задачи.
2. На странице — секция `## Связанные Beads`.
3. В метаданных bead — `wiki_refs` на эту страницу.
4. Дописать запись в `log.md`.

Тривиальные правки кода (опечатки, форматирование) обновления вики и
decision-страниц не требуют.

## Перед коммитом/пушем — проверить актуальность

Перед коммитом убедись, что изменения кода отражены в вики. Это же напоминает
git-хук `commit-msg` (`.beads/hooks/commit-msg`) — он срабатывает на любой
коммит в репозитории (терминал, IDE, агент): если в коммите есть изменения кода
без правок `.claude/wiki/`, печатает предупреждение в stderr. Коммит при этом
проходит — дисциплину держит это правило и команда `/wiki-ingest`, а не хук.
Способы устранить предупреждение:

- обновить вику и добавить её файлы в коммит (вручную или командой
  `/wiki-ingest` — см. ниже);
- если изменение правда не требует правок вики (тривиальная правка) — добавить в
  сообщение коммита трейлер `Wiki: skip` (подавит предупреждение).

## Догон вики по дельте кода — `/wiki-ingest`

Команда `/wiki-ingest` (`.claude/commands/wiki-ingest.md`) — наполнение вики
по уже сделанным правкам кода: читает дельту (`git diff`/ref-range/аргументы),
сопоставляет затронутые файлы с страницами по `sources:` и slug'ам, обновляет
или создаёт страницы, правит `index.md` и `log.md`. Применяется в первую
очередь, когда правки делались вне Claude Code (через IDE/терминал) и вика
отстала.

## Периодическая проверка — `/wiki-lint`

Команда `/wiki-lint` (`.claude/commands/wiki-lint.md`) — ручной воркфлоу
проверки целостности вики и синхронизации с beads: битые `[[…]]`,
страницы-сироты, консистентность `wiki_refs` и `[[bead:…]]`, отражение
закрытых задач в `decisions/`. В отличие от `/wiki-ingest`, не читает дельту
кода — проверяет согласованность уже написанных страниц.

## Формат log.md

Хронологический, append-only. Каждая запись — с единообразным префиксом:

```
## [2026-05-22] ingest | Заголовок изменения
## [2026-05-22] query  | Текст вопроса
## [2026-05-22] lint   | Краткий итог проверки
```
